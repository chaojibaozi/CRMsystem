## 修复 CSV 导入乱码问题（decodeGBK 函数）

**问题描述：** 数据管理模块导入存量数据 CSV 时报"未匹配到任何列"，解析到的表头为韩文乱码。

**根本原因：** `decodeGBK` 函数（约第 3561 行）是手写的伪 GBK 解码器，错误地将 GBK 双字节序列的原始字节值 `(byte1 << 8) | byte2` 直接当作 Unicode code point 处理，导致所有中文字符均解码为乱码。

**修复方案：** 替换为基于浏览器原生 `TextDecoder` API 的自动编码检测函数：
1. 优先检测 UTF-8 BOM（EF BB BF）
2. 尝试 `fatal: true` 模式 UTF-8 解码，成功则判定为 UTF-8
3. 失败则调用 `new TextDecoder('gbk')` 使用浏览器内置 GBK 码表解码
4. 最终兜底 `gb2312`

影响位置：`initCustomerListImport`、`initCustomerPoolImport`、`initStockCustomerImport` 三处 CSV 分支均调用此函数，一次修复全部生效。

## 修复"没有新客户可导入"问题（IMPORT_FIELD_MAP 别名补充 + 行对齐）

**问题描述：** CSV 解码正常、预览显示正确，但点击确认导入提示"没有新客户可导入（可能手机号已存在或姓名为空）"。

**根本原因（两个）：**
1. **别名缺失**：IMPORT_FIELD_MAP 只有"姓名"/"联系人"两个 name 字段别名，用户 CSV 表头为"客户姓名"等变体，无法匹配，导致 `customer.name` 未被赋值，触发姓名为空跳过逻辑。

**修复内容：**
- 三处 IMPORT_FIELD_MAP 均补充：`客户姓名`、`客户名称`、`联系人姓名`、`公司`、`手机号码`、`电话`、`微信账号`、`联系邮箱`、`地址`、`客户来源`、`客户等级`、`下次跟进时间` 等常见别名。
- 三处 `json.length < 2` 检查之后均插入行对齐代码：将数据行列数补齐到表头列数，避免 undefined 取值。

## 修复 CSV GBK 编码自动检测（detectAndParseWorkbook）

**问题描述：** Windows 中文版 Excel 导出的 CSV 文件使用 GBK 编码，而 SheetJS 的 `XLSX.read()` 对 CSV 默认按 UTF-8/Latin-1 猜测编码，导致中文字符全部乱码（如 `ÕË»§Ãû³Æ`），无法匹配任何列名。

**修复方案：** 新增 `detectAndParseWorkbook(file)` 异步函数，基于三条件自动编码检测：
1. 检测 `\uFFFD` 替换字符 → 非 UTF-8，回退 GBK
2. 检测 Latin 扩展字符（U+00C0–U+024F）占比 → GBK 被误判为 Latin-1，回退 GBK
3. 正常 UTF-8 解码 → 保留原结果
4. 通过 `XLSX.read(text, { type: 'string' })` 绕过 SheetJS 的编码猜测

影响范围：三个导入入口（客户列表、客户池、存量数据）全部替换 `readAsArrayBuffer` + `XLSX.read(type:'array')` 为 `await detectAndParseWorkbook(file)` + `XLSX.read(type:'string')`，与原有 `IMPORT_FIELD_MAP` 匹配逻辑完全兼容。
